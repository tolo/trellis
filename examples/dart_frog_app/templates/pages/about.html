About This Example

Provider-Based Dependency Injection

Dart Frog uses providers for dependency injection. The Trellis engine is registered once in _middleware.dart via trellisProvider(_engine) and automatically available to all route handlers through context.read<Trellis>(). The renderPage and renderFragment helpers read the engine from the context automatically.

File-Based Routing

Routes map directly to files in the routes/ directory. routes/index.dart handles GET /, routes/about.dart handles GET /about, and the counter mutations live in routes/counter/. Shared in-memory state sits in lib/counter_state.dart, and _middleware.dart applies middleware to all routes in its directory and below.

Middleware Chain

Middleware is composed via .use() calls in _middleware.dart. This example chains: Trellis engine provider, security headers, CSRF protection, request logging, and (in dev mode) template hot reload. Order matters — security middleware runs before route handlers.

HTMX Fragment Rendering

HTMX requests are detected via the HX-Request header using isHtmxRequest(context). When detected, handlers return only the relevant HTML fragment instead of the full page. Navigation links use hx-get with hx-target="#content" and hx-push-url="true" for SPA-style page transitions. The counter buttons use hx-post to replace only the counter fragment.

CSRF Protection

The trellisCsrf() middleware implements double-submit cookie CSRF protection. A csrfToken variable is automatically injected into the template context. The base layout stores it in a <meta> tag, and a small script sends it as an X-CSRF-Token header on every HTMX request. Forms include a hidden _csrf field as a fallback.

Template Hot Reload

When the DEV=true environment variable is set, trellis_dev's devMiddleware watches template files and injects an SSE script. Template changes are reflected instantly in the browser without restarting the server. Run with DEV=true dart_frog dev to enable.
